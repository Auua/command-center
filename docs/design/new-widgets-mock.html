Load balancing: what the balancer actually knows
6-day streak

A round-robin balancer knows nothing about your servers except that they answered the last health check. Least-connections knows how busy they are. That difference is the whole lesson.

  1. Client sends HTTPS requests to Balancer.
  2. Balancer forwards requests to App 1.
  3. Balancer forwards requests to App 2.
  4. Balancer polls App 1 and App 2 with health checks (asynchronous).
  5. App 1 and App 2 read from Shared cache.
Load balancing strategies compared
Strategy Upside Cost
Round robin Stateless, trivially correct, no per-server bookkeeping. One slow server keeps getting its share of traffic.
Least connections Routes away from a struggling node automatically. Balancer must track live connection counts per node.
Consistent hashing Same key lands on the same node — cache locality. Hot keys become hot nodes; rebalancing is a real event.
Self-check: why must a health check be cheaper than the request it guards?
Because the balancer runs it against every node on every interval. If the check does real work, you have built a self-inflicted load generator that gets heavier exactly when the fleet is sick.
